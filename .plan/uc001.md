# UC-001: 참가 신청 구현 계획

## 1. 유스케이스 개요

**행위자**: 번역버디
**목적**: 봉사활동에 참여 신청을 하여 봉사시간을 쌓기 위함
**인터페이스**: 카카오톡 챗봇
**엔드포인트**: `POST /kakao/api/applications`

## 2. 비즈니스 요구사항

### 2.1 핵심 규칙
- **BR-001**: 하나의 학번은 중복 참가 불가
- **BR-002**: 이름은 2~4글자 허용
- **BR-003**: 학번 형식은 XX-XXX (두 자리 연도, 세 자리 번호)
- **BR-004**: 승인되지 않은 사용자는 시스템 이용 불가

### 2.2 페이즈 제약
- 모집(RECRUIT) 또는 번역(TRANSLATE) 페이즈에서만 신청 가능

### 2.3 기본 흐름
1. 번역버디가 학번(XX-XXX)과 이름(2~4글자)을 입력하여 신청 제출
2. 시스템이 학번 중복 여부 확인
3. [중복 시] 오류 메시지 표시 후 종료
4. 시스템이 학번 형식과 이름 길이 검증
5. [형식 불일치 시] 오류 메시지 표시 후 종료
6. 시스템이 승인 대기 상태의 신청을 저장
7. 성공 메시지 표시

#### 3.2.4 Service: `ApplicationService`
**위치**: `application/service/application/ApplicationService.kt`

**의존성**:
- `LoadApplicationPort` (신청 조회)
- `SaveApplicationPort` (신청 저장)
- `LoadSystemPhasePort` (시스템 페이즈 조회, 이미 구현됨)
- `Clock` (현재 시간)

**핵심 로직**:
```kotlin
@Transactional
override fun applyParticipation(command: ApplyParticipationCommand): ApplicationResult {
    // 1. 시스템 페이즈 확인
    val systemPhase = loadSystemPhasePort.load() ?: throw InvalidPhaseException("...")
    if (systemPhase.systemPhaseType !in listOf(SystemPhaseType.RECRUIT, SystemPhaseType.TRANSLATE)) {
        throw InvalidPhaseException("...")
    }

    // 2. 학번 값 객체 생성 및 검증
    val studentNumber = StudentNumber(command.studentNumber)
    studentNumber.validate()

    // 3. 학번 중복 확인
    if (loadApplicationPort.existsByStudentNumber(studentNumber)) {
        throw DuplicateStudentNumberException("...")
    }

    // 4. 신청 엔티티 생성 및 검증
    val now = Instant.now(clock)
    val application = Application.create(studentNumber, command.name, now)
    application.validate()

    // 5. 저장
    val saved = saveApplicationPort.save(application)

    // 6. 결과 반환
    return ApplicationResult(
        studentNumber = saved.studentNumber.value,
        name = saved.name,
        status = saved.status,
        appliedAt = saved.appliedAt
    )
}
```

### 3.3 Adapter Layer

#### 3.3.1 Inbound HTTP Adapter
**위치**: `adapter/inbound/application/http/`

**`ApplicationApiController.kt`**:
```kotlin
@RestController
@RequestMapping("/kakao/api/applications")
class ApplicationApiController(
    private val applyParticipationUseCase: ApplyParticipationUseCase
) {
    @PostMapping
    fun applyParticipation(
        @RequestBody request: ApplyParticipationRequest
    ): ApplicationResponse {
        val command = ApplyParticipationCommand(
            studentNumber = request.studentNumber,
            name = request.name
        )
        val result = applyParticipationUseCase.applyParticipation(command)
        return ApplicationResponse(
            studentNumber = result.studentNumber,
            name = result.name,
            status = result.status,
            appliedAt = result.appliedAt

    }
}
```

**`ApplicationJpaAdapter.kt`**:
```kotlin
@Repository
class ApplicationJpaAdapter(
    private val applicationJpaRepository: ApplicationJpaRepository
) : LoadApplicationPort, SaveApplicationPort {

    override fun existsByStudentNumber(studentNumber: StudentNumber): Boolean {
        return applicationJpaRepository.existsByStudentNumber(studentNumber.value)
    }

    override fun save(application: Application): Application {
        val jpaEntity = ApplicationJpaEntity(
            studentNumber = application.studentNumber.value,
            name = application.name,
            status = application.status,
            appliedAt = application.appliedAt,
            rejectionReason = application.rejectionReason,
            processedAt = application.processedAt
        )
        val saved = applicationJpaRepository.save(jpaEntity)
        return application.copy(
            // id가 필요하다면 추가 필드 고려
        )
    }

    private fun ApplicationJpaEntity.toDomain(): Application {
        return Application(
            studentNumber = StudentNumber(studentNumber),
            name = name,
            status = status,
            appliedAt = appliedAt,
            rejectionReason = rejectionReason,
            processedAt = processedAt
        )
    }
}
```

  - 잘못된 페이즈에서의 신청 시나리오

- **POST /kakao/api/applications**:

  - 중복 학번 (409 Conflict)
  - 잘못된 페이즈 (403 Forbidden)

## 6. 주의사항

### 6.1 트랜잭션 처리
- `ApplicationService.applyParticipation()` 메서드에 `@Transactional` 적용
- 학번 중복 확인과 저장 사이의 경합 조건(Race Condition) 방지를 위해 DB unique constraint 활용

### 6.2 예외 처리
- 모든 도메인 예외는 `SofiaException` 상속
- `GlobalExceptionHandler`에서 일관된 에러 응답 처리

### 6.3 페이즈 검증
- `LoadSystemPhasePort`를 통해 현재 시스템 페이즈 조회
- null인 경우 기본값(INACTIVE) 처리 고려

### 6.4 향후 확장성
- UC-002(신청 승인/거절) 구현 시 `rejectionReason`, `processedAt` 필드 활용
- 승인된 신청자는 별도 `Participant` 엔티티로 변환하는 로직 필요 (추후 고려)

## 7. 의존성 관계

```
Domain Layer
    ↓ (의존)

    ↓ (구현)
```
